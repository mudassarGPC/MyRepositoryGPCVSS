# Supabase Database Setup Guide - My Portfolio

This document explains the SQL commands used to set up and fix the `subscribers` table for the email gate integration. These commands were run in the Supabase SQL Editor.

## 1. Setting up Permissions (The "Access" Fix)
If you see an error like `42501: permission denied`, it means the database is blocking "anonymous" visitors from writing to the table.

```sql
GRANT USAGE ON SCHEMA public TO anon;
GRANT INSERT ON TABLE public.subscribers TO anon;
```
*   **Why we ran this:** By default, Supabase roles are very restrictive. These commands tell the database: "It is okay to let visitors (who aren't logged in) 'see' the public area and 'add' new rows to the subscribers table."

---

## 2. Fixing the ID Column (The "Auto-Pilot" Fix)
If the database complains about a missing `id`, it's because it expects you to provide a number for every new row. We want the database to do this automatically.

```sql
-- Remove the old identity setting
ALTER TABLE public.subscribers ALTER COLUMN id DROP IDENTITY IF EXISTS;

-- Re-add it correctly as "Generated by Default"
ALTER TABLE public.subscribers ALTER COLUMN id ADD GENERATED BY DEFAULT AS IDENTITY;
```
*   **Why we ran this:** This turns the `id` column into an "Identity" column. It tells the database: "Whenever a new email comes in, please look at the last ID used and automatically assign the next number (1, 2, 3...)." This prevents errors where the `id` was left empty.

---

## 3. Fixing the Timestamp Constraint (The "Simultaneous Sign-up" Fix)
By default, some tables might mistakenly require every sign-up time to be unique.

```sql
ALTER TABLE public.subscribers DROP CONSTRAINT IF EXISTS subscribers_created_at_key;
```
*   **Why we ran this:** If two people clicked the button at the exact same millisecond, the "Unique" rule would block the second person. This command removes that restriction, allowing multiple people to sign up at the same time without issues.

---

## 4. Enabling Row Level Security (The "Security Gate")
Even with the permissions above, Supabase uses "Policies" to double-check every action.

**In the Supabase Dashboard (not SQL):**
1.  Enable **RLS** on the `subscribers` table.
2.  Add a policy: **"Allow anonymous inserts"**.
3.  Set operation to **INSERT**, role to **anon**, and the check expression to `true`.

*   **Why we did this:** RLS is Supabase's main security layer. The policy acts like a "whitelist" rule that specifically allows the `INSERT` action for your visitor role.

---

### Future Reference
If you ever create a new table (e.g., for a contact form or guestbook), you will likely need to follow these same steps:
1.  **Grant** permissions to `anon`.
2.  Set the ID to **Identity** (Generated by Default).
3.  Add an **RLS Policy** for `INSERT`.
